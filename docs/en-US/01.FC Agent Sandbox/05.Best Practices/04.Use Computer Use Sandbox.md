# Use Computer Use Sandbox (Invite-Only Preview)

> This feature is currently in internal invitational testing. The content may change based on test results.

Computer Use Sandbox is designed for scenarios where an Agent needs to operate a computer the way a person would. It is suitable for tasks that require a desktop environment, browser or application windows, screen observation, mouse and keyboard input, and result evaluation. Unlike Browser Use Sandbox, Computer Use does not only control web pages — it controls an isolated Linux desktop session that can operate the browser, terminal, file manager, editor, and business applications simultaneously.

Based on the current reference implementation, the correct usage pattern is not to give the model an unrestricted desktop. Instead, prepare a `desktop` template that includes an XFCE desktop, browser, screenshot capability, mouse and keyboard control, file and command capabilities, and then let the business-side Agent Harness establish an "observe, decide, restricted execute, audit" closed loop.

If the task only involves web page access, clicking, screenshots, and downloads, prefer [Browser Use Sandbox](02.Use%20Browser%20Use%20Sandbox.md). If Python/Node processing is needed after browser operations in the same session, prefer [AIO Sandbox](03.Use%20AIO%20Sandbox.md). Only use Computer Use Sandbox when the task genuinely requires cross-desktop applications, terminal, file system, or human remote takeover.

## Use Cases

- Automate enterprise back-office systems to complete cross-page data entry, screenshots, and file exports.
- Reproduce UI issues reported by users and generate an action trace.
- Let an Agent complete combined browser, terminal, and file-processing tasks in an isolated environment.
- Evaluate desktop-operation Agents while recording every observation, action, and result.
- Enable human takeover in remote desktops for login, CAPTCHA, or high-risk confirmations, then hand control back to the Agent.

## Recommended Architecture

Computer Use should be integrated as follows:

1. The business system defines the task, allowed applications, allowed domains, and operation boundaries.
2. Create a sandbox from a prepared `desktop` template. The template is typically based on an AIO custom runtime with built-in XFCE, browser, desktop control service, and necessary tools.
3. The Agent Harness waits for the desktop to be ready, obtains the screen size and initial screenshot.
4. Pass the screenshot, task goal, and action history to the vision model, and let the model return restricted actions.
5. The Harness validates actions and only allows explicit actions such as `click`, `double_click`, `right_click`, `type/write`, `press`, `scroll`, `drag`, `wait`, `screenshot`, etc.
6. High-risk actions such as login, form submission, payment, file deletion, message sending, and configuration changes go through human approval.
7. Take a screenshot after each step and record observations, actions, window state, model output, and result files.
8. Export the trace, screenshots, and artifacts at the end of the task, then destroy the sandbox.

## Prepare a Desktop Template

The current Computer Use reference implementation uses a desktop template rather than a regular browser template. The template should include at least:

- Ubuntu 22.04 or an equivalent Linux base environment.
- XFCE or another desktop environment.
- Applications to be operated: browser, terminal, file manager, editor, etc.
- Desktop control service supporting screenshots, mouse, keyboard, windows, files, and commands.
- Optional remote desktop or livestream capability for debugging and human takeover.

During development and evaluation, you can use prebuilt or internal preview images to create AIO custom runtime templates. Production environments should be consolidated into controlled templates — do not write account credentials, cookies, or long-term tokens into the image.

## Example Code

The following example demonstrates the basic control loop of the Desktop SDK: create a desktop sandbox, wait until ready, take a screenshot, execute restricted actions, take another screenshot, and destroy the sandbox. When integrating a vision model in practice, pass the `screenshot`, screen size, and action history to the model, and let the model return the next action.

> **Note**: Computer Use is in invite-only preview and the Desktop SDK has not been officially released yet. The `your_sandbox_sdk`, `DesktopSandbox`, `<desktop-sdk-package>` etc. below are placeholder names to illustrate the control loop and **cannot be copied and run directly**. Use the actual Desktop SDK package name and API when it becomes available.

```python
import logging
import os
from pathlib import Path

from your_sandbox_sdk import DesktopSandbox

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

TEMPLATE_NAME = os.getenv("SANDBOX_TEMPLATE_NAME") or os.getenv("TEMPLATE_NAME")


def save_screenshot(desktop: DesktopSandbox, path: str) -> None:
    image = desktop.screenshot()
    Path(path).write_bytes(image)
    logger.info("saved screenshot: %s (%s bytes)", path, len(image))


desktop = DesktopSandbox.create(
    template_name=TEMPLATE_NAME,
    wait_timeout=120,
    idle_timeout_seconds=600,
)

try:
    logger.info("sandbox created: %s", desktop.sandbox_id)

    health = desktop.check_health()
    logger.info("health: %s", health)

    width, height = desktop.get_screen_size()
    logger.info("screen size: %sx%s", width, height)

    save_screenshot(desktop, "01_desktop_ready.png")

    # In production, the vision model should return actions based on the screenshot.
    # This example uses fixed actions to demonstrate the Computer Use observe-act-observe loop.
    desktop.launch("xfce4-terminal")
    desktop.wait(2000)
    save_screenshot(desktop, "02_terminal_opened.png")

    desktop.write("echo hello from computer use sandbox")
    desktop.press("enter")
    desktop.wait(1000)
    save_screenshot(desktop, "03_command_result.png")

finally:
    desktop.kill()
```

For installation and runtime, the business side needs to configure access credentials and template name:

```bash
pip install <desktop-sdk-package> python-dotenv

export FC_ACCESS_KEY_ID=your-access-key-id
export FC_ACCESS_KEY_SECRET=your-access-key-secret
export FC_ACCOUNT_ID=your-main-account-id
export TEMPLATE_NAME=your-desktop-template

python example_desktop.py
```

## Remote Observation and Human Takeover

If developers or users need to observe the desktop in real time, you can start a desktop stream. Production environments must enable authentication — do not expose unauthenticated desktops.

```python
from your_sandbox_sdk import DesktopSandbox

desktop = DesktopSandbox.create(
    template_name="your-desktop-template",
    wait_timeout=120,
    idle_timeout_seconds=600,
)

try:
    desktop.stream.start(require_auth=True)
    auth_key = desktop.stream.get_auth_key()
    url = desktop.stream.get_url(auth_key=auth_key)
    print(f"Desktop stream: {url}")

    # Execute Agent tasks here, or wait for humans to complete
    # login, CAPTCHA, or high-risk confirmations.
    desktop.wait(60_000)

finally:
    try:
        desktop.stream.stop()
    finally:
        desktop.kill()
```

If the template does not provide a `stream` capability, you can also start a VNC or noVNC service inside the template and expose the access address through a sandbox port. Regardless of the method used, authentication, time-limited validity, and access logging should be enforced.

## Action Space Design

Constrain the model output to structured actions rather than letting the model generate system commands directly:

```json
{
  "action": "click",
  "x": 320,
  "y": 240,
  "reason": "focus the terminal input area"
}
```

The Harness should validate before execution:

- Coordinates must be within the current screen size.
- Text input should be length-limited and obviously dangerous content should be filtered.
- `press` should only allow whitelisted shortcuts, such as `enter`, `escape`, `tab`, `ctrl+a`, `ctrl+c`.
- `commands.run` should only be used for business-side explicitly authorized diagnostic or preparation steps, not as free-form model actions.
- A screenshot must be taken after each round of actions and the execution result recorded.

## Production Recommendations

- Do not accept arbitrary Shell commands generated by the model. Desktop actions and command execution should be managed with separate permissions.
- Screenshots may contain sensitive information. Apply masking or access control before sending them to the model.
- Login state, cookies, and account credentials should be injected per task on a temporary basis — do not write them into the template.
- Login, payment, deletion, message sending, and configuration change actions must go through human approval.
- Record every observation, action, screenshot, window state, model output, and artifact for review and audit.
- Set task timeouts, maximum step counts, idle timeouts, and automatic cleanup policies.
- Desktop tasks cost more resources. Use Computer Use only when Browser/AIO cannot meet the requirements.
