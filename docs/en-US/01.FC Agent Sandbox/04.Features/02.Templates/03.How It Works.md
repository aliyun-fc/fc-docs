# How It Works

Templates capture the runtime environment that must be prepared before a sandbox starts. Instead of reinstalling dependencies after every sandbox is created, you move that work into the template build phase.

## Lifecycle

1. **Define** — describe the runtime environment with the `Template` builder, starting from a base image.
2. **Build** — submit the definition with `Template.build()`. The platform produces a template image and returns a template ID and build ID.
3. **Ready** — once the build status becomes `ready`, the template can be used to create sandboxes.
4. **Create** — create a sandbox from the template by template ID or name. Each sandbox is an independent runtime instance.

## Relationship between templates and sandboxes

- A template defines the base environment.
- A sandbox is a runtime instance created from a template.
- Multiple sandboxes can use the same template.
- Updating a template does not automatically change sandboxes that are already running.

## What belongs in a template

- System dependencies and language runtimes.
- Package manager dependencies such as Python and Node.js packages.
- Common CLI tools.
- Base code and configuration for business agents.

## What should not go into a template

- User input data.
- Temporary files for a single task.
- Business state that changes frequently.
- Secrets, tokens, and other sensitive credentials.
